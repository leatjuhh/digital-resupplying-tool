# Sessie 23 Maart 2026 - Consolidatieprogressie en Git Push Prep

Deze sessielog bundelt de daadwerkelijk afgeronde progressie van vandaag en zet de repository in een commit-ready vorm voor een volgende git-push. Dit document is historisch; de leidende actuele waarheid blijft in `docs/technical/current-state.md` en `todo/master-backlog.md`.

## Waarom deze sessie push-relevant is

De consolidatieslag van vandaag rondt drie eerder open slices inhoudelijk af en legt daarnaast de lokale start- en herstelworkflow expliciet vast:

- settings en RBAC zijn nu aangesloten op echte backenddata
- dashboard en overzichten zijn nu data-gedreven
- baseline fase 1 draait in shadow mode mee
- de lokale dev launcher kan nu hangende servers schoon herstarten via `.\dev.ps1 -Restart`

## Wat vandaag inhoudelijk is afgerond

### 1. Settings en RBAC op echte backenddata

- `/settings` is nu permission-driven in plaats van rol- of placeholder-gedreven
- algemene instellingen en regels laden en bewaren nu echte backendwaarden
- users CRUD gebruikt echte rollen via `role_id`
- er is een expliciete `Rollen`-tab voor permissionbeheer
- API-key beheer is write-only richting frontend en toont alleen masked metadata terug

Praktisch effect:

- admins zien alle settingsdomeinen
- gewone users zien alleen toegestane tabs of read-only gedrag
- store-users zien `Instellingen` niet meer en worden bij directe toegang omgeleid

### 2. Dashboard nu data-gedreven

- er is een auth-beschermde dashboard summary endpoint toegevoegd
- dashboardkaarten tonen echte KPI's in plaats van samplewaarden
- wachtende reeksen, recente systeemevents en aandachtspunten komen uit echte backenddata
- de periodeselectie doet geen nepvergelijkingen meer, maar communiceert expliciet dat het om een live totaaloverzicht gaat
- het proposals-overzicht is opgeschoond: geen samplefilters of misleidende fake-acties meer

Praktisch effect:

- het dashboard voelt nu als een echt overzichtsscherm en niet langer als een placeholder
- gebruikers kunnen via echte links door naar proposal- en assignmentflows

### 3. Baseline-algoritme fase 1

- proposals krijgen nu een stabiele `Situation: ...` marker in `applied_rules`
- de classifier onderscheidt `LOW_STOCK`, `MEDIUM_STOCK`, `HIGH_STOCK` en `PARTIJ`
- dit draait bewust in shadow mode en verandert de move-generatie nog niet
- er is een offline evaluatiehaak toegevoegd voor lokale geĂŻmporteerde weekdata

Praktisch effect:

- het algoritme levert al bruikbare classificaties zonder de bestaande proposalflow te verstoren
- de handmatige herverdelingen van de twee lokaal beschikbare weken kunnen nu tegen de classifier worden afgezet

### 4. Assignments-slice bevestigd in pushset

- assignments draaien op echte backenddata en zijn store-facing uitvoerbaar
- de smoke-suite dekt store-login en assignmentsnavigatie af

Praktisch effect:

- goedgekeurde proposals hebben nu een echte uitvoeringslaag voor winkelgebruikers

### 5. Lokale restartflow via `dev.ps1`

- `.\dev.ps1 -Restart` is toegevoegd als officiĂ«le lokale herstelroute
- het script stopt processen op poort `3000` en `8000` en start daarna backend en frontend opnieuw
- quick start en GUI-debugdocs verwijzen nu expliciet naar dit restartpad

Praktisch effect:

- een vastgelopen vorige frontend- of backendsessie hoeft niet meer handmatig buiten de launcher om te worden beĂ«indigd

## Bevestigde validatie van deze sessie

- `frontend`: `npm run build` geslaagd
- `backend`: relevante pytest-set voor dashboard, users, baseline en datamapping geslaagd
- `repo root`: `npm run browser:smoke` geslaagd op een stabiele frontendstart

Belangrijk detail:

- de frontend devserver gaf tussentijds een `.next`/chunk-cacheprobleem rond `/login`
- de uiteindelijke smoke-validatie is daarom bevestigd tegen een schone frontendstart, zodat de functionele code en niet de vervuilde dev-cache werd gevalideerd

## Bekende huidige randvoorwaarden

- baseline fase 1 blijft shadow mode; ranking, selectie en optimizer gebruiken de situatie nog niet
- OpenAI-key-validatie is nog mock-validatie en nog niet encrypted-at-rest
- het proposals-overzicht heeft nu echte sortering, maar nog geen echte backendfiltering of zoekquery
- assignments en settings zijn functioneel echt, maar blijven buiten de leidende productkern

## Git Push Prep

### Werkboom logisch reviewen

Gebruik deze drie reviewgroepen voordat je staged:

- app/functioneel
  backend routes, services, modellen en tests
  frontend pagina's, components, API-contracten en smoke tests
  `dev.ps1`
- leidende docs
  `README.md`
  `docs/technical/current-state.md`
  `docs/getting-started/*`
  `todo/master-backlog.md`
  `AGENTS.md`
  `docs/PROJECT_CONTEXT_INDEX.md`
- archief/consolidatie
  `archive/2026-03-consolidation/`
  deletions uit `docs/` en `todo/` die daarin zijn ondergebracht

### Commit-ready structuur

Aanbevolen commitvolgorde:

1. `docs: finalize session and consolidation docs`
2. `feat: wire settings dashboard assignments and baseline slices`
3. `chore: add dev launcher restart flow`

Compacter alternatief:

- `docs: finalize session and consolidation docs`
- `feat: wire settings dashboard assignments baseline and dev restart`

### Pre-push checks

Voer vlak voor commit of push minimaal opnieuw uit:

- `frontend`: `npm run build`
- `backend`: relevante pytest-set voor dashboard/settings/baseline/users
- `repo root`: `npm run browser:smoke`
- optioneel: `git diff --check`
- optioneel: `git status --short`

Controleer expliciet dat deze niet mee staged:

- `frontend/next-dev.log`
- `.next/`
- Playwright artifacts
- lokale testoutput
- overige tijdelijke runtimebestanden

## Korte pushsamenvatting

- settings/RBAC is nu permission-driven op echte backenddata
- dashboard draait nu data-gedreven via een auth-beschermde summary endpoint
- baseline fase 1 voegt situatieclassificatie in shadow mode plus offline evaluatie toe
- `.\dev.ps1 -Restart` maakt lokale serverrestart bij bezette poorten onderdeel van de standaard workflow
